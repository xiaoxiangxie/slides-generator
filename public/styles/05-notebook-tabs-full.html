S01 · 核心闭环

这一章要解决
什么问题

语言模型本身只会"生成下一段内容"。它不会自己：

打开文件

运行命令

观察报错

把工具结果再接着用于下一步推理

核心目标：把"模型 + 工具"连接成一个能持续推进任务的主循环。

S01 · 关键概念定义

4 个核心概念

loop

"只要任务还没做完，系统就继续重复同一套步骤。"

turn

发消息 → 读回复 → 执行工具 → 写回结果，一轮完整的推理周期。

tool_result

"工具结果必须重新进入消息历史、成为下一轮推理的输入。"

state

"当前运行状态" — 主循环继续往下走时，需要一直带着走的那份数据。

关键数据结构

"消息历史不是聊天记录展示层，而是模型下一轮要读的工作上下文。"

S01 · 最小实现步骤

第一步：
准备初始消息

messages = [{"role": "user", "content": query}]

用用户的查询启动初始消息列表，这是整个循环的起点。

从一条用户消息开始，模型就能在此基础上进行第一轮推理。

S01 · 最小实现步骤

第二步：
调用模型

response = client.messages.create(
model = MODEL,
system = SYSTEM,
messages = messages,
tools = TOOLS,
max_tokens = 8000,
)

把当前整个消息历史发给模型，让它决定下一步动作。

S01 · 最小实现步骤

第三步：
追加 assistant 回复

messages.append({
"role": "assistant",

})

很多初学者会忽略这一步。

只关心"最后有没有答案"，忽略把 assistant 回复本身写回历史。

S01 · 完整循环代码

完整循环代码
(45行)

def agent_loop(state): while True: response = client.messages.create( model = MODEL, system = SYSTEM, messages = state["messages"], tools = TOOLS, max_tokens = 8000, ) state["messages"].append({ "role": "assistant", "content": response.content, }) if response.stop_reason != "tool_use": state["transition_reason"] = None return results = [] for block in response.content: if block.type == "tool_use": output = run_tool(block) results.append({ "type": "tool_result", "tool_use_id": block.id, "content": output, }) state["messages"].append( {"role": "user", "content": results} ) state["turn_count"] += 1 state["transition_reason"] = "tool_result"
S01 · 核心闭环

为什么先
接受简化

初学者在第一章真正要学会的，不是所有复杂边界，而是：

assistant 回复要写回历史

tool_result 要写回历史

主循环要持续推进

流式、并发、恢复、压缩这些高级内容，留给后续章节。

S01 · 核心闭环

最容易
犯的错

❌ 工具结果打印出来，但不写回 messages

模型下一轮根本看不到真实执行结果

❌ 只保存用户消息，不保存 assistant 消息

上下文会断层

❌ 不给工具结果绑定 tool_use_id

模型会分不清哪条结果对应哪次调用

❌ 一上来就塞进流式、并发、恢复、压缩

第一章还消化不了这些

❌ 以为 messages 只是聊天展示

在 agent 里，messages 更像"下一轮工作输入"，而不仅仅是聊天记录

S01 · 核心闭环

一句话
记住

Agent Loop 的本质，
是把"模型的动作意图"变成
"真实执行结果"，
再把结果送回模型继续推理。
